# 위젯 전수 점검 보고서 (v0.2.3 기준)

작성일: 2026-05-22
대상: 디자인 위젯 (Electron 앱) + GAS 백엔드
독자: 비개발자 오너(의사결정용)

---

## 한 줄 요약

**v0.2.3에서 알림 폭주 버그는 해결됐어. 다만 "디자이너가 시트의 마감일을 바꾸면 같은 작업이 가짜 NEW로 다시 뜨는 문제", "디자이너가 실수로 시트 구조(헤더 행, 열 순서)를 바꾸면 위젯 전체가 어긋나는 문제", "사용자가 위젯을 끄지 않으면 자동 업데이트가 영원히 안 깔리는 문제" 세 개가 가장 신경 쓸 만한 약점이야.**

## 1. 즉시 결정해야 할 것 (우선순위 높음)

### 1-1. 마감일 바꾸면 가짜 NEW 뜸

| 항목 | 내용 |
|---|---|
| **언제 일어나** | 디자이너가 시트에서 어떤 작업의 마감일을 변경할 때 (예: 5/22 → 5/23) |
| **사용자에게 뭐가 보임** | 이미 본 작업인데 위젯에 NEW가 다시 표시됨. OS 토스트 알림도 한 번 더 발사 |
| **운영 빈도 추정** | 자주. 디자인팀 시트에서 마감일 변경은 흔함 (월 10회 이상?) |
| **영향 강도** | 약함. 그 작업 하나만 NEW로 다시 뜨고 끝. 전체 폭주 아님 |
| **왜 발생하나** | 위젯의 식별 키가 "마감일+광고주+비고". 마감일이 키 안에 들어 있어서 마감일 변경 = 새 작업 |
| **고침 옵션 A** | 키에서 마감일 빼고 "광고주+비고"로. 단 같은 광고주의 같은 비고가 다른 마감일로 두 번 나오는 케이스 충돌 (v0.2.0 김수현 버그 재발) |
| **고침 옵션 B** | 키는 "마감일+광고주+비고" 유지하되, 마감일이 변경된 행을 자동 인지해서 "원래 봤던 것"으로 처리. 복잡함 |
| **추천** | **당장 안 고침**. 이번 v0.2.3 안정화 후 운영하면서 디자인팀 반응 보고 결정. 불만 안 나오면 그대로 두기 |

### 1-2. 시트 구조 변경 → 위젯 전체 어긋남

| 항목 | 내용 |
|---|---|
| **언제 일어나** | 디자이너가 시트 9행(헤더 행) 위에 행을 추가하거나, E~L 열 순서를 바꾸거나, 작업자 컬럼(F) 위치를 옮겼을 때 |
| **사용자에게 뭐가 보임** | 위젯이 엉뚱한 데이터를 보여줌 (예: 광고주 자리에 수량이 표시), 또는 상태 변경 클릭이 다른 행을 바꿔버림 |
| **운영 빈도 추정** | 거의 없음. 디자인팀은 시트 구조 안 건드림. 단, 실수 가능성 0은 아님 |
| **영향 강도** | 심각. 잘못된 행이 변경되면 데이터 손상 |
| **왜 발생하나** | GAS 코드가 "광고주는 5번째 열, 작업자는 6번째 열" 식으로 고정 위치를 하드코딩 |
| **고침 옵션 A** | GAS가 첫 행(헤더)에서 "광고주" "작업자" 텍스트를 찾아 동적으로 위치 결정. 견고함 |
| **고침 옵션 B** | 현 시트 구조에 "수정 금지" 표시만 디자인팀에게 안내. 코드 변경 0 |
| **추천** | **옵션 B (안내만)**. 6명 작은 팀, 디자인팀 시트는 구조 잘 안 건드림. 옵션 A는 작업량 대비 효용 낮음 |

### 1-3. 자동 업데이트가 영원히 안 깔리는 케이스

| 항목 | 내용 |
|---|---|
| **언제 일어나** | 새 버전이 다운로드됐는데 사용자가 위젯을 한 번도 안 끄면 |
| **사용자에게 뭐가 보임** | 아무것도 안 보임. 위젯은 계속 구 버전. 새 버전 있다는 안내 없음 |
| **운영 빈도 추정** | 자주. 디자이너들은 위젯을 켜놓고 며칠~몇 주씩 안 끔. PC 절전만 함 |
| **영향 강도** | 중간. 버그 수정·신기능 배포가 느려짐. 김수현 같은 케이스에서 fix가 뒤늦게 도달 |
| **왜 발생하나** | electron-updater의 자동 설치는 "앱 종료 시"에만 실행. 사용자에게 알림 없음 |
| **고침 옵션 A** | 다운로드 완료 시 위젯에 "새 버전 준비됨, 재시작 권장" 토스트 추가. 클릭하면 즉시 재시작·설치 |
| **고침 옵션 B** | 1주일 지나면 자동으로 위젯 재시작 (사용자 동의 없이). 위험 |
| **추천** | **옵션 A**. 작업량 적고 효용 큼. 다음 fix 배포 사이클에 포함 권장 |

## 2. 가까운 시일 내 결정 (우선순위 중간)

### 2-1. 빠른 연속 클릭 → 토스트 5개 폭주

- **언제**: 사용자가 상태 버튼(미정→대기→진행 등)을 1초 내 여러 번 클릭
- **현 동작**: 클릭마다 토스트 발사. 큐는 직렬이라 데이터는 안전하지만 UI는 시끄러움
- **빈도**: 가끔
- **영향**: 약함 (불편 정도)
- **고침**: 버튼 클릭 후 0.5초간 비활성화. 작업량 작음
- **추천**: **다음 사이클 fix 시 같이**

### 2-2. 같은 행을 두 명이 동시 변경

- **언제**: 두 디자이너가 동시에 같은 작업의 상태를 다른 값으로 변경
- **현 동작**: 마지막에 도착한 변경이 이긴다 (last-write-wins). 둘 다 성공 응답 받음
- **빈도**: 거의 없음 (다른 사람의 작업을 건드릴 일 자체가 드뭄)
- **영향**: 중간. 한 명이 "내가 진행으로 바꿨는데?" 혼란
- **고침**: 충돌 감지 후 두 번째 변경을 거부 (현 expect 검증 로직 강화)
- **추천**: **놔두기**. 운영상 거의 없는 케이스. 위젯이 5분마다 자동 새로고침해서 어차피 곧 일치 인식

### 2-3. 다른 팀원이 공유 처리 중인데 내가 그 행 상태 변경

- **언제**: A가 공유 체크 누른 순간 B가 같은 행의 상태를 클릭
- **현 동작**: GAS가 "시트가 변경됨" 에러 → 위젯이 자동 재시도. 그래도 안 되면 "변경 실패" 토스트
- **빈도**: 가끔
- **영향**: 약함. 토스트로 사용자에게 보여줌
- **고침**: 토스트 문구를 더 친근하게 ("다른 팀원이 처리했어요" 등)
- **추천**: **다음 사이클 UI 정리 시 같이**

### 2-4. 시트에 작업자 이름 오타

- **언제**: 디자이너가 작업자 칸에 "김수현" 대신 "김수형" 등 입력
- **현 동작**: 그 행은 아무 위젯에도 안 보임 (유령 작업)
- **빈도**: 거의 없음
- **영향**: 중간. 누군가의 작업이 누락
- **고침**: GAS에 "목록에 없는 작업자 이름" 경고 로그 추가. 또는 시트에 작업자 드롭다운 강제
- **추천**: **시트에 드롭다운 검증 추가**. 코드 0줄, 디자인팀이 시트에서 한 번 설정

### 2-5. 디자이너가 비고 오타 수정

- **언제**: "메타01" → "메타 01" 같은 소소한 수정
- **현 동작**: 그 행 키 변경 → 가짜 NEW로 다시 표시
- **빈도**: 자주
- **영향**: 약함 (그 행만)
- **고침**: 마감일 변경(1-1)과 본질 같음. 동시 해결 가능
- **추천**: **1-1과 같이 묶어서 결정**

## 4. 안전하다고 판단되는 영역 (걱정 안 해도 됨)

- 네트워크·재시도·캐시 처리: 견고함
- 멤버 전환 (다른 디자이너 보기): 깨끗하게 동작
- 위젯 창·트레이·키보드 단축키: 잘 정리됨
- 데이터 영속화 (electron-store): 안전
- 빠른 연속 작업의 데이터 정합성: 큐 직렬화로 보호 (UI 노이즈만 약점)

## 6. 이번 점검에서 발견된 v0.2.3 잔여 약점 한 줄 정리

> "마감일·비고가 수정되면 그 행 하나만 가짜 NEW로 다시 뜸" + "사용자가 위젯 안 끄면 자동 업데이트 안 깔림" + "시트 구조 변경 대응 약함(거의 안 일어남)" 세 개.

전체 폭주, 데이터 손상, 다중 사용자 충돌 같은 심각한 위험은 v0.2.3에서 깨끗함. 가짜 NEW 한 건 정도는 운영 가능한 수준.

# DATA_CONTRACT.md — 공유 시트 데이터 계약

> **이 문서를 먼저 읽어라.** 시트 구조·컬럼·GAS URL을 건드리는 모든 작업은 이 문서를 먼저 확인하고 시작한다.
> 한 시트를 세 프로그램(위젯 / 대시보드 / 입력기[예정])이 공유하므로, 한쪽 변경이 다른 쪽을 조용히 깨뜨릴 수 있다.
> **작성 기준**: 2026-05-25 / 위젯 레포(`design-widget-schedule`) + 대시보드 레포(`design-dashboard`) 코드 대조.

---

## 0. 한 줄 요약

데이터는 **신규·유지보수 → 완료 → 업무데이터** 한 방향으로만 흐른다.
앞단 두 시트(신규·유지보수, 완료)는 **열 위치**로 다뤄지고(위젯·자동화 GAS), 끝단(업무데이터)은 **헤더 이름**으로 읽힌다(대시보드).
그래서 위험의 종류가 다르다 — 앞단은 *열을 옮기면* 깨지고, 끝단은 *헤더 이름을 바꾸면* 깨진다.

---

## 1. 데이터 흐름

```
┌─ 위젯 파일 (1-zGcP9ao6qGjNXRAijxU9rOar88bHIZHw_HKleBc2Ys) ─┐   ┌─ 업무데이터 파일 (1KukGD6…) ─┐
│                                                          │   │                            │
│ [입력기(예정)]      [위젯]                               │   │      [대시보드]            │
│      │ 쓰기      읽기＋쓰기                               │   │      │ 읽기 전용           │
│      ▼          ▼      ▼                                 │   │      ▼                     │
│  신규·유지보수 ──공유 시──▶ 완료 ────매일 9시 동기화──────┼───┼──▶ 업무데이터              │
│    (열 위치)            (열 위치)                         │   │    (헤더 이름)             │
│        └──── legacy-gas 자동화 ────┘                      │   │                            │
└──────────────────────────────────────────────────────────┘   └────────────────────────────┘
                                    Sync가 파일 경계를 넘어 데이터를 넘긴다
```

- **신규·유지보수 → 완료**: 디자이너가 공유 처리하면 `Scheduler.gs`(legacy-gas)가 행을 완료 시트로 이동. (같은 파일 내)
- **완료 → 업무데이터**: 매일 오전 9시 `Synccompletedtodatasheet.gs`가 완료 행을 **다른 파일**의 업무데이터로 이관(ID 전파 포함). Sync의 `DATA_SHEET_ID`에 업무데이터 파일 ID가 박혀 있다.
- **두 시트 파일은 물리적으로 분리돼 있다.** 위젯 파일에서 사람이 뭘 하든 업무데이터 파일은 직접 영향받지 않는다 — 오직 Sync라는 다리를 거쳐서만.
  - 장점: 앞단 사고가 대시보드로 즉시 번지지 않음.
  - 단점: **Sync 하나가 끊기면 대시보드가 통째로 옛 데이터에 멈춘다 (단일 지점).** 그리고 앞단 변경은 다음 9시 동기화를 거쳐야 대시보드에 보인다.

## 3. 누가 어떻게 읽나 (위험의 뿌리)

| 프로그램 | 보는 시트 | 식별 방식 | 깨지는 조건 |
|---|---|---|---|
| 위젯 (`schedule-widget-api.gs`) | 신규·유지보수, 완료 | **열 위치** (E·F·L… 하드코딩) | 열을 삽입/이동하면 즉사 |
| 자동화 (`Scheduler.gs`, `Synccompletedtodatasheet.gs`) | 신규·유지보수, 완료 → 업무데이터 | **열 위치** | 〃 |
| 대시보드 (`gas/dashboard.gs`) | 업무데이터 | **헤더 이름** | 헤더명을 바꾸면 즉사 (열 이동엔 안전) |

핵심: **앞단과 끝단의 안전 규칙이 정반대다.** 앞단은 "열 순서 고정"이 생명, 끝단은 "헤더 이름 고정"이 생명.

## 4. 위험 규칙 (변경 전 반드시 확인)

1. **업무데이터 헤더 이름 변경 → 대시보드 즉사.** 연도/월/팀/광고주/작업유형/수량/TAT/완료일 중 하나라도 이름이 달라지면 끝. (열은 옮겨도 됨)
2. **신규·유지보수 / 완료 시트에 열 삽입·이동 → 위젯 + 자동화 GAS 즉사.** 위치 하드코딩이라 한 칸만 밀려도 어긋남. → GAS를 동시에 고쳐야 하고, 디자인팀 휴무 중에만 진행. (v0.2.4 L열 ID 신설이 이 케이스였음)
3. **L열 UUID 손상/삭제 → 위젯 식별 깨짐(가짜 NEW 폭주) + Sync 중복체크 깨짐.** 절대 사람이 직접 만지지 않는다.
4. **앞단 변경은 대시보드에 즉시 반영되지 않는다.** 완료→업무데이터 동기화(매일 9시)를 거쳐야 보인다.
5. **시트는 사람이 직접 편집하는 공간이다 = 가장 약한 지점.** 드래그 실수로 셀 한 칸 어긋나도 위 1~3이 발동할 수 있음. → 사람 통제(권한·안내·Ctrl+Z)로 막는 게 현재 정책.

## 5. GAS 운영 규칙

1. **`.gs` 파일의 git 버전이 원본(source of truth).** GAS 콘솔에서 코드를 고쳤으면 **즉시 레포에도 반영**한다. (콘솔만 고치고 레포 반영을 빠뜨리면 → 클코가 레포의 옛 코드를 믿고 작업하다 어긋남)
2. **재배포로 GAS URL이 바뀌면, 위젯과 대시보드 둘 다 교체한다.**
   - 위젯: `electron/main.js`의 `GAS_BASE` (확인 완료)
   - 대시보드: `config.ts`의 `GAS_URL` (※ 변수명 별도 확인 필요 — §6 참조)
   - 한쪽만 바꾸면 다른 쪽이 옛 GAS를 **조용히** 본다 (가장 알아채기 어려운 사고).
3. **대시보드 캐시**: `gas/dashboard.gs`는 시트를 최대 1시간 캐싱한다. 시트 수정 후 즉시 반영하려면 `invalidateCache()` 호출 또는 `CACHE_VERSION` 상수를 올린다.
   - (참고: 현재 운영자는 GAS를 매번 새 URL로 재배포 → 사실상 캐시가 초기화되어 이 문제를 자동 우회 중)
4. GAS 웹앱 접근 권한은 "모든 사용자"(익명 fetch 필요).
